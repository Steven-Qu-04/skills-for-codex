
# Arrange Mode 工程设计审核报告

## 一、总体结论

当前工程设计整体 **通过方向审查**。

它已经正确处理了几个核心问题：

1. 没有把多个知识库粗暴合并成一个大图。
2. 没有把每本书降级成一个 book-level 节点。
3. 明确要求 verified cross-edge 必须有 child verified paths。
4. Read Mode 已经考虑 `single_kb` / `cross_kb` 分支。
5. 保留了 `one_sided`、`hypothesis`、`needs_review`、`rejected` 等降级状态。
6. 明确禁止用 cross node summary、semantic payload、route report 作为最终证据。
7. 保持了脚本薄入口 + `router_core.py` 统一分发的工程风格。

这说明设计已经基本符合 Arrange Mode 的核心哲学：

> Arrange Mode 负责跨库建桥；Read Mode 负责沿桥查询；每条桥都必须能回到两端 child KB 的证据路径。

但在进入实现前，还需要修正一些契约模糊点，否则 MVP 实现时容易出现“候选关系很多，但 verified cross-edge 很少”“cross index 有结构但 Read Mode 不好用”“child query 任务生成了但未形成闭环”等问题。

---

## 二、主要优点

### 1. 模式定位清晰

设计明确区分：

```text
Construct Mode：构建单库
Read Mode：读取单库 / 跨库索引
Arrange Mode：构建跨库编排层
```

这是正确的。Arrange Mode 不替代 Construct，也不替代 Read，而是位于多个已构建知识库之上，生成跨库索引。

### 2. 证据边界清楚

设计中多次强调：

```text
verified cross-edge 必须有双侧 child verified paths
cross answer 必须引用 cross evidence path、child verified path、child source span
cross index 不复制 child atoms / nodes / edges 全量内容
```

这避免了跨库索引退化成“摘要式综合”或“无证据知识图谱”。

### 3. Read Mode 适配方向正确

`query_route.py` 增加：

```text
--scope single_kb | cross_kb | auto
--cross-index
--kb-registry
--cross-edges
--cross-evidence-paths
```

并新增：

```text
cross_synthesis
cross_concept_trace
cross_conflict_trace
cross_evidence_verification
```

这是必要的。否则 Arrange Mode 即使构建出 cross index，Read Mode 也无法真正消费它。

### 4. MVP 风险控制合理

设计明确允许：

```text
cross_communities.jsonl 可 stubbed
部分候选 needs_review
query_child_kbs.py 第一版可只生成 child query tasks
```

这能避免第一版被自动化验证拖死。

---

## 三、需要修正的关键问题

## P0-1：`query_child_kbs.py` 第一版如果只生成任务，会削弱 verified cross-edge 闭环

当前设计说第一版 `query_child_kbs.py` 可以只生成 child query tasks，并把无法验证者标记为 `needs_review`。这对快速落地合理，但与质量门槛中的：

```text
verified_cross_edges_with_two_sided_evidence / verified_cross_edges >= 0.95
```

存在张力。

如果第一版不自动执行 child KB Read Mode，那么大部分候选都只能停留在 `needs_review`，Arrange Mode 可能只能产出骨架，难以产出真正 verified cross-edge。

### 建议修订

明确分成两个 profile：

```text
arrange_skeleton：
  只发现 KB、生成候选、生成 child query tasks、生成 stub cross index。
  不要求 verified cross-edge 数量。

arrange_mvp：
  至少对高优先级候选自动调用 child KB Read Mode。
  可以生成少量 verified cross-edge。
```

质量门槛也应分开：

```text
arrange_skeleton:
  required_files_exist = true
  child_query_tasks_generated > 0
  verified_cross_edges 可以为 0，但 artifact_status = partial

arrange_mvp:
  valid_child_kbs >= 2
  verified_cross_edges >= 1
  verified_cross_edges_with_two_sided_evidence / verified_cross_edges >= 0.95
```

否则 `arrange_mvp` 的验收标准会过高，而第一版实现能力又偏低。

---

## P0-2：缺少 `child_query_tasks.jsonl` 产物契约

设计中说 `query_child_kbs.py` 第一版可生成 child query tasks，但输出产物中没有明确列出：

```text
child_query_tasks.jsonl
```

这会导致 “任务生成”和 “任务执行结果” 混在一起。

### 建议新增

Arrange output 增加：

```text
child_query_tasks.jsonl
child_query_sessions.jsonl
```

其中：

```json
{
  "child_query_task_id": "cqt_<hash>",
  "alignment_candidate_id": "",
  "kb_id": "",
  "question": "",
  "mode": "local_reading | concept_trace | argument_trace | source_verification",
  "target_objects": [],
  "required_evidence_types": [],
  "status": "pending | executed | failed | skipped | needs_review"
}
```

这样 Arrange Mode 可以清楚区分：

```text
candidate → child query task → child route session → cross evidence path → cross edge
```

---

## P0-3：`alignment_candidates.jsonl` 不应允许 `verified` 状态

当前设计写：

```text
status=candidate | needs_child_query | verified | rejected | hypothesis
```

但 alignment candidate 是候选，不应直接进入 verified。verified 应该属于 `cross_edges.jsonl` 或 `verified_alignment_records.jsonl`。

### 建议修订

`alignment_candidates.jsonl` 的状态改为：

```text
candidate | needs_child_query | queried | promoted | rejected | hypothesis | needs_review
```

其中：

```text
promoted = 已被提升为 cross_edge
verified = 只能用于 cross_edge.verification_status
```

这样避免候选层和正式边层混淆。

---

## P0-4：Read Mode 的 `--scope auto` 需要明确判定规则

设计新增了：

```text
--scope single_kb | cross_kb | auto
```

但没有说明 `auto` 如何判断。实现时容易误判，尤其当用户同时提供 `--index` 和 `--cross-index`。

### 建议规则

```text
1. 显式 --scope 优先。
2. 如果传入 --cross-index，则默认 cross_kb。
3. 如果只传入 --index，则默认 single_kb。
4. 如果 workspace 中存在 arrange_output/cross_route_index.sqlite，且问题包含“不同知识库 / 多本书 / 比较 / 冲突 / 综合 / 跨文档”等意图，auto 选择 cross_kb。
5. 如果无法判断，报错并要求显式传 --scope。
```

不要让 auto 静默选择，避免跨库问题被单库回答。

---

## P0-5：缺少 cross route path 的评分标准

`route_scoring.md` 计划增加 cross path scoring，但工程设计没有定义最低字段。

Arrange Mode 的路径评分不能直接复用单库评分，因为它要考虑：

```text
双侧证据
子库覆盖
冲突保留
跨库边可靠性
child path 质量
```

### 建议新增 cross path score

```json
{
  "cross_path_score": {
    "query_relevance": 0.0,
    "cross_edge_reliability": 0.0,
    "two_sided_evidence": 0.0,
    "child_path_quality": 0.0,
    "kb_coverage": 0.0,
    "conflict_preservation": 0.0,
    "evidence_directness": 0.0,
    "redundancy_penalty": 0.0,
    "hypothesis_penalty": 0.0,
    "one_sided_penalty": 0.0,
    "final_score": 0.0
  }
}
```

并规定：

```text
cross_conflict_trace 优先保留 contrasts/refutes/qualifies 边；
cross_synthesis 优先保留多 KB 覆盖；
cross_source_verification 优先保留最短 verified child path。
```

## 四、P1 级改进建议

### P1-1：SKILL.md description 可能过长

当前 description 信息完整，但较长，可能影响触发精度。建议将关键触发词前置，压缩为：

```yaml
description: Construct, arrange, and read source-verifiable knowledge routing networks for long-form documents. Use for books, PDFs, reports, archives, or workspaces of constructed KBs when Codex must create anchored Markdown, source spans, route graphs, cross-KB indexes, or answer through verified single-KB/cross-KB route paths instead of top-k chunks. Do not use for short summaries, simple OCR, ordinary format conversion, or ordinary vector RAG.
```

详细触发条件放进 `SKILL.md` 正文即可。

---

### P1-2：`cross_node` 的用途需要进一步收窄

cross node 容易变成“跨书主题摘要”。建议明确：

```text
cross_node 是跨库路由入口，不是最终证据，也不是融合后结论。
```

并要求：

```text
cross_node.member_objects 不得为空
cross_node.summary 必须保留来源差异
cross_node 不能单独支撑 final answer
```

---

### P1-3：`build_alignment_candidates.py` 需要候选预算策略

自动候选可能爆炸。设计提到了风险，但脚本职责中还没有预算参数。

建议增加 CLI：

```bash
--max-candidates-per-kb-pair 200
--max-candidates-per-type 100
--min-label-score 0.3
--min-payload-score 0.2
```

并输出：

```text
candidate_generation_report.json
```

报告包括：

```text
candidate_count_by_kb_pair
candidate_count_by_type
dropped_by_budget
dropped_by_low_score
```

---

### P1-4：`validate_arrange.py` 需要检查“书籍节点化”

设计在失败模式里提到防护，但没有写入质量门槛。

建议新增门槛：

```text
book_level_only_cross_edges = 0
cross_edges_with_object_level_target / total_cross_edges >= 0.95
```

并明确：

```text
cross_edge.source_object.type / target_object.type 不得只为 book | kb | document
```

---

### P1-5：`query_child_kbs.py` 应该保留 child query prompt

为了可审计，建议每次 child query 记录：

```json
{
  "child_query_task_id": "",
  "kb_id": "",
  "question": "",
  "mode": "",
  "target_objects": [],
  "prompt_or_query_payload": {},
  "created_from_alignment_candidate": "",
  "status": ""
}
```

否则后续无法判断跨库边是如何验证出来的。

## 五、P2 优化建议

### P2-1：增量 Arrange

成熟版提到 `add_new_kb`，但可以提前预留字段：

```json
{
  "arrange_run_type": "initial | incremental_add_kb | incremental_reverify"
}
```

这样后续加入新知识库时，不必重建全部 cross index。

### P2-2：区分 source authority / source perspective

不同书的权威性、体裁、时间、作者立场可能不同。可以在 `kb_registry.jsonl` 中预留：

```json
{
  "source_metadata": {
    "author": "",
    "publication_year": null,
    "discipline": "",
    "source_role": "primary | secondary | commentary | manual | dataset | unknown"
  }
}
```

这对冲突解释很有用。

### P2-3：跨库 conflict map

成熟版可单独输出：

```text
cross_conflicts.jsonl
```

字段：

```json
{
  "conflict_id": "",
  "topic": "",
  "positions": [],
  "supporting_cross_edges": [],
  "evidence_paths": [],
  "status": "open | resolved_by_scope | unresolved | needs_review"
}
```

## 六、建议修订后的最小实现边界

当前“推荐最小实现边界”里包含 `query_route.py --scope cross_kb`，这是对的。但还应补上：

```text
child_query_tasks.jsonl
arrange_validation_report.json
candidate_generation_report.json
```

推荐第一版闭环改为：

```text
1. arrange_protocol.md
2. discover_kbs.py
3. build_alignment_candidates.py
4. child_query_tasks.jsonl generation
5. verify_cross_edges.py
6. build_cross_route_index.py
7. validate_arrange.py
8. query_route.py --scope cross_kb
```

其中：

```text
query_child_kbs.py 可以第一版只生成 child_query_tasks；
但 arrange_mvp 不能声称有 verified cross-edge，除非至少部分 child queries 已执行并产生 child verified paths。
```

---

## 七、最终审查结论

这份工程设计已经基本达到可实现水平，尤其在以下方面表现很好：

```text
模式定位清楚
证据边界清楚
Read Mode 适配方向正确
脚本分层合理
失败模式有意识
不退化为 top-k RAG
```

但实现前必须修正以下 5 点：

```text
1. 增加 child_query_tasks.jsonl 契约。
2. 将 alignment candidate 的 verified 状态移到 cross edge 层。
3. 明确 --scope auto 判定规则。
4. 增加 cross path scoring 最低字段。
5. 区分 arrange_skeleton 与 arrange_mvp，避免“只生成任务”却声称完成 verified arrange。
```

最终建议：

> 设计评审：条件通过。
> 修完 P0 五项后，可进入 Arrange Mode MVP 实现。
> 第一版重点不是追求大量跨库边，而是保证每条 verified cross-edge 都有双侧 child verified paths，并且 Read Mode 能沿 cross evidence path 回到 child source spans。
