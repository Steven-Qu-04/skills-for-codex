# longform-knowledge-router Arrange Mode 工程设计

## 1. 背景与目标

本设计以根目录 `arrange_mode-standard.md` 为技术标准，为 `longform-knowledge-router` 增加第三种运行模式：`Arrange Mode`。

现有 skill 已支持：

- `Construct Mode`：从原始长文构建单个可验证知识库。
- `Read Mode`：读取单个已构建知识库，通过 route path 和 source span 回答问题。

新增 `Arrange Mode` 后，skill 需要支持：

- 在一个 workspace 中发现多个已构建 child KB。
- 为多个 child KB 建立跨知识库编排层。
- 生成跨库节点、跨库边、跨库证据路径和 cross route index。
- 让 Read Mode 能读取 cross route index，并在回答中回到 child KB 的 verified paths 和 source spans。

一句话目标：

> Arrange Mode 建桥，Read Mode 走桥；桥的两端必须落到各自 child KB 的原文证据。

## 2. 设计原则

本工程按 `skill-creator` 的技能设计原则执行：

1. **保持 SKILL.md 精简**  
   `SKILL.md` 只放模式选择、主流程、必须读取的 reference、核心命令和硬规则。详细对象、schema、验证规则进入 `references/arrange_protocol.md` 和 `references/schemas/`。

2. **渐进加载上下文**  
   只有用户触发 Arrange Mode 时，才要求读取 `references/arrange_protocol.md`。只有涉及跨库 Read Mode 时，才读取 cross read 相关段落和 schema。

3. **脚本负责确定性工作，AI 负责语义判断**  
   脚本可发现 KB、校验文件、生成候选、写索引、执行结构验证；不得把 label/embedding 相似度直接当作 verified cross-edge。

4. **保留证据边界**  
   Arrange Mode 不复制 child KB 全图，不复制全文，只保存跨库关系和对子库证据路径的引用。

5. **分歧是一等公民**  
   不把冲突强行综合为单一结论；必须保留 `contrasts_with`、`refutes`、`qualifies`、`one_sided`、`hypothesis`、`needs_review` 等状态。

## 3. 工程范围

### 3.1 本次新增能力

- 增加 Arrange Mode 的 skill 入口说明。
- 增加 Arrange Mode 的协议 reference。
- 增加 Arrange Mode 输出 schema。
- 增加 Arrange Mode 脚本入口。
- 扩展 `router_core.py` 的命令分发。
- 扩展 Read Mode，使其识别 `single_kb` 与 `cross_kb`。
- 增加 Arrange Mode 验证报告。
- 增加 child query task、candidate generation report 与 cross path scoring 契约。
- 更新 README 使用说明。

### 3.2 非目标

- 不实现多库全文合并。
- 不做统一 top-k chunk 检索。
- 不把多个 `route_graph.json` 机械拼接成一个大图。
- 不在没有 child verified paths 的情况下生成 verified cross-edge。
- 不把 cross node summary、semantic payload、route report 当作最终证据。

## 5. SKILL.md 改造设计

### 5.1 description 更新

当前 description 只覆盖 construct/read 单库。需要加入跨知识库编排触发语义：

```yaml
description: Construct, arrange, and read source-verifiable knowledge routing networks for long-form documents. Use for books, PDFs, reports, archives, or workspaces of constructed KBs when Codex must create anchored Markdown, source spans, route graphs, cross-KB indexes, or answer through verified single-KB/cross-KB route paths instead of top-k chunks. Do not use for short summaries, simple OCR, ordinary format conversion, or ordinary vector RAG.
```

详细触发条件放入正文的 Mode Decision，frontmatter description 保持短而高召回。

### 5.2 Mode Decision 更新

新增决策规则：

```text
Use Arrange Mode when the user provides a workspace containing multiple constructed output directories, multiple route_index.sqlite files, or asks to compare, align, synthesize, contrast, verify, or build an index across multiple existing knowledge bases.
```

### 5.3 新增 Arrange Mode 小节

`SKILL.md` 中新增简洁流程，不展开完整对象定义：

```bash
python scripts/discover_kbs.py --workspace <workspace_dir> --output-dir <arrange_output>
python scripts/build_alignment_candidates.py --kb-registry <arrange_output>/kb_registry.jsonl --output-dir <arrange_output>
python scripts/query_child_kbs.py --alignment-candidates <arrange_output>/alignment_candidates.jsonl --kb-registry <arrange_output>/kb_registry.jsonl --output-dir <arrange_output>
python scripts/verify_cross_edges.py --alignment-candidates <arrange_output>/alignment_candidates.jsonl --cross-evidence-paths <arrange_output>/cross_evidence_paths.jsonl --output-dir <arrange_output>
python scripts/build_cross_route_index.py --cross-edges <arrange_output>/cross_edges.jsonl --cross-evidence-paths <arrange_output>/cross_evidence_paths.jsonl --output-dir <arrange_output>
python scripts/validate_arrange.py --arrange-output <arrange_output> --profile arrange_skeleton | arrange_mvp
```

硬规则写入 `SKILL.md`：

- verified cross-edge 必须有双侧 child verified paths。
- `same_as` 必须有强对齐证据，不得只靠名称相同。
- cross index 不复制 child atoms/nodes/edges 全量内容。
- cross answer 必须引用 cross evidence path、child verified path、child source span。

## 6. Arrange Mode 数据流

```text
workspace
→ discover child KBs
→ validate child KBs
→ write arrange_manifest.json and kb_registry.jsonl
→ load registries / route reports / semantic payloads as navigation aids
→ generate alignment_candidates.jsonl
→ generate child_query_tasks.jsonl
→ query child KB Read Mode for evidence when profile requires verification
→ write child_query_sessions.jsonl
→ verify alignments and cross edges
→ write cross_nodes.jsonl and cross_edges.jsonl
→ write cross_evidence_paths.jsonl
→ build cross_route_templates.jsonl and cross_route_index.sqlite
→ validate arrange output
→ write arrange_report.md
```

关键约束：

- `alignment_candidates.jsonl` 是候选，不是最终边。
- `cross_edges.jsonl` 中 `verification_status=verified` 的记录必须双侧有证据。
- `cross_route_index.sqlite` 只索引跨库导航入口、关系和 evidence path，不承载 child KB 全文。

### 7.1 arrange_manifest.json

用途：记录 arrange run 的全局配置、输入 workspace、输出状态和 profile。

必要字段：

```json
{
  "artifact_status": "complete | partial | failed",
  "created_at": "",
  "workspace": "",
  "output_dir": "",
  "profile": "arrange_skeleton | arrange_mvp | arrange_mature",
  "kb_count": 0,
  "valid_kb_count": 0,
  "commands": [],
  "arrange_run_type": "initial | incremental_add_kb | incremental_reverify",
  "quality_gates": {}
}
```

### 7.2 kb_registry.jsonl

用途：记录 child KB 身份、路径、可查询能力和验证状态。

必要规则：

- 没有 `route_index.sqlite` 的 KB 不能作为 verified child KB。
- 缺少 `source_spans.jsonl` 的 KB 不能生成 verified cross-edge。
- 缺少 `concept_registry.jsonl` 或 `claim_registry.jsonl` 时可进入 limited arrange，但候选质量降低。
- 预留 `source_metadata.author`、`publication_year`、`discipline`、`source_role`，用于后续解释来源权威性与立场差异。

### 7.3 alignment_candidates.jsonl

用途：保存跨库候选关系。

必要规则：

- `status=candidate | needs_child_query | queried | promoted | rejected | hypothesis | needs_review`。
- 候选 basis 可以来自 label、registry、route report、payload、embedding 或模型判断。
- `verified` 只能出现在 `cross_edges.jsonl` 的 `verification_status`，不得作为 alignment candidate 状态。
- `promoted` 表示候选已被提升为正式 cross edge；候选本身仍不是证据。

### 7.4 cross_nodes.jsonl

用途：保存跨库路由入口。cross node 不是最终证据，也不是融合后结论。

必要规则：

- `cross_node.member_objects` 不得为空。
- `cross_node.summary` 必须保留来源差异、立场差异和适用范围差异。
- cross node 不能单独支撑 final answer；最终回答必须继续落到 cross evidence path 和 child source spans。

### 7.5 cross_edges.jsonl

用途：保存正式跨库边。

必要规则：

- `verification_status=verified` 时，`source_evidence_paths` 与 `target_evidence_paths` 都不能为空。
- `relation_type=same_as` 必须通过强证据校验。
- `hypothesis` edge 可用于探索，不能作为最终回答证据。

### 7.6 cross_evidence_paths.jsonl

用途：保存跨库查询的证据路径。

必要规则：

- 每条 path 至少包含一个 cross step。
- verified path 必须包含不少于两个 child paths，除非任务本身是单侧定位并明确标注。
- child path 必须可回到 child route session、verified path 和 source spans。

### 7.7 child_query_tasks.jsonl

用途：把“需要验证的候选关系”拆成可审计、可执行的 child KB 查询任务，避免任务生成和任务执行结果混在一起。

必要字段：

```json
{
  "child_query_task_id": "cqt_<hash>",
  "alignment_candidate_id": "",
  "kb_id": "",
  "question": "",
  "mode": "local_reading | concept_trace | argument_trace | source_verification",
  "target_objects": [],
  "required_evidence_types": [],
  "prompt_or_query_payload": {},
  "created_from_alignment_candidate": "",
  "status": "pending | executed | failed | skipped | needs_review"
}
```

状态流：

```text
alignment candidate
→ child query task
→ child route session
→ cross evidence path
→ cross edge
```

### 7.8 报告类产物

`candidate_generation_report.json` 记录候选生成预算和丢弃原因：

```text
candidate_count_by_kb_pair
candidate_count_by_type
dropped_by_budget
dropped_by_low_score
```

`arrange_validation_report.json` 记录 profile、质量门槛、失败项、降级项和是否可声明完成。
## 8. 脚本设计

现有脚本多为：

```python
from router_core import main
main("<command>")
```

新增 Arrange Mode 脚本延续该模式，减少入口分散。

### 8.1 discover_kbs.py

命令：

```bash
python scripts/discover_kbs.py --workspace <workspace_dir> --output-dir <arrange_output>
```

职责：

- 递归扫描 workspace。
- 识别包含 `route_index.sqlite`、`route_graph.json`、`source_spans.jsonl` 的 child KB。
- 读取 `build_report.md`、`validation_report.json`、`concept_registry.jsonl`、`claim_registry.jsonl` 的存在状态。
- 写 `arrange_manifest.json`、`kb_registry.jsonl`、`errors.jsonl`。

不做：

- 不生成跨库关系。
- 不读取 child KB 全文。

### 8.2 build_alignment_candidates.py

命令：

```bash
python scripts/build_alignment_candidates.py --kb-registry <kb_registry.jsonl> --output-dir <arrange_output> --max-candidates-per-kb-pair 200 --max-candidates-per-type 100 --min-label-score 0.3 --min-payload-score 0.2
```

职责：

- 从 child KB registry、concept registry、claim registry、route nodes、route reports 中生成候选。
- 计算基础相似特征：label overlap、alias overlap、payload term overlap、source type overlap。
- 写 `alignment_candidates.jsonl`、`cross_concept_registry.jsonl`、`cross_claim_registry.jsonl`、`candidate_generation_report.json`。

约束：

- 输出默认状态为 `candidate` 或 `needs_child_query`。
- 候选预算由 `--max-candidates-per-kb-pair`、`--max-candidates-per-type`、`--min-label-score`、`--min-payload-score` 控制。
- 不得直接写 `cross_edges.jsonl`。

### 8.3 query_child_kbs.py

命令：

```bash
python scripts/query_child_kbs.py --alignment-candidates <alignment_candidates.jsonl> --kb-registry <kb_registry.jsonl> --output-dir <arrange_output>
```

职责：

- 对需要验证的候选生成 child KB Read Mode 查询任务。
- 写 `child_query_tasks.jsonl`，记录 prompt/query payload、目标对象和 required evidence types。
- 在 `arrange_mvp` 中，至少对高优先级候选调用或复用 `query_route.py` 的单库能力，生成 child route sessions。
- 整理 child verified paths、source spans、uncertainties。
- 写 `child_query_sessions.jsonl`、`cross_evidence_paths.jsonl` 初稿。

MVP 实现建议：

- `arrange_skeleton` 可以只生成 query task，并把无法验证者标记为 `needs_review`。
- `arrange_mvp` 不能只停留在 query task，必须至少执行一部分高优先级 child queries，并在存在双侧证据时产出 verified cross-edge。
- 对高置信、低风险候选可调用现有 `query_route.py --mode concept_trace/local_reading`。

### 8.4 verify_cross_edges.py

命令：

```bash
python scripts/verify_cross_edges.py --alignment-candidates <alignment_candidates.jsonl> --cross-evidence-paths <cross_evidence_paths.jsonl> --output-dir <arrange_output>
```

职责：

- 对候选关系执行四级验证：alignment、child path、cross-edge、cross-answer readiness。
- 写 `cross_edges.jsonl`、`rejected_alignments.jsonl`、`repair_suggestions.jsonl`。

降级规则：

- 单侧证据：`one_sided`。
- 只有模型推断：`hypothesis`。
- 子库查询失败：`needs_review`。
- 关系类型不被证据支持：`rejected` 或降级为更弱关系。

### 8.5 build_cross_route_index.py

命令：

```bash
python scripts/build_cross_route_index.py --cross-edges <cross_edges.jsonl> --cross-evidence-paths <cross_evidence_paths.jsonl> --output-dir <arrange_output>
```

职责：

- 聚合 `cross_nodes.jsonl`。
- 生成 `cross_route_templates.jsonl`。
- 构建 `cross_route_index.sqlite`。
- 写 `cross_communities.jsonl`，MVP 可标记 `artifact_status=stubbed`。

SQLite 建议表：

```sql
cross_nodes(cross_node_id, type, title, summary, verification_status, payload_json)
cross_edges(cross_edge_id, source_kb_id, target_kb_id, relation_type, edge_class, verification_status, confidence, payload_json)
cross_evidence_paths(cross_evidence_path_id, question_or_intent, answer_affordance, verification_status, payload_json)
node_terms(cross_node_id, term)
edge_terms(cross_edge_id, term)
kb_registry(kb_id, title, kb_path, validation_status, payload_json)
```

### 8.6 validate_arrange.py

命令：

```bash
python scripts/validate_arrange.py --arrange-output <arrange_output> --profile arrange_skeleton | arrange_mvp
```

职责：

- 检查必需文件。
- 检查 verified edge 的双侧证据。
- 检查 `same_as` 是否有强对齐证据。
- 检查 cross index 是否没有复制 child 全量图。
- 检查 rejected alignment 是否被记录。
- 写 `arrange_validation_report.json`。

## 10. Read Mode 适配设计

### 10.1 CLI 扩展

`query_route.py` 当前要求：

```text
--question
--index
--mode local_reading | source_location_query | concept_trace
--strategy explicit_path | graph_activation | hybrid
--session-dir
```

新增：

```text
--scope single_kb | cross_kb | auto
--kb <kb_id>
--workspace <workspace_dir>
--cross-index <cross_route_index.sqlite>
--kb-registry <kb_registry.jsonl>
--cross-edges <cross_edges.jsonl>
--cross-evidence-paths <cross_evidence_paths.jsonl>
```

新增 mode：

```text
cross_local_reading
cross_synthesis
cross_concept_trace
cross_argument_trace
cross_conflict_trace
cross_evidence_verification
cross_source_verification
```

兼容策略：

- 显式 `--scope` 优先。
- 如果传入 `--cross-index`，则默认 `cross_kb`。
- 如果只传入 `--index`，则默认 `single_kb`。
- 如果 `--scope auto` 且 workspace 中存在 `arrange_output/cross_route_index.sqlite`，并且问题包含“不同知识库 / 多本书 / 比较 / 冲突 / 综合 / 跨文档”等意图，选择 `cross_kb`。
- 如果 `--scope auto` 无法判断，报错并要求显式传 `--scope`；不要静默选择。
- 保留旧 `--index` 用法不变。

### 10.3 Cross Path Scoring

跨库路径评分不能直接复用单库路径评分。`score_paths.py` 或 cross read 分支需要为 candidate cross path 输出最低字段：

```json
{
  "cross_path_score": {
    "query_relevance": 0.0,
    "cross_edge_reliability": 0.0,
    "two_sided_evidence": 0.0,
    "child_path_quality": 0.0,
    "kb_coverage": 0.0,
    "conflict_preservation": 0.0,
    "evidence_directness": 0.0,
    "redundancy_penalty": 0.0,
    "hypothesis_penalty": 0.0,
    "one_sided_penalty": 0.0,
    "final_score": 0.0
  }
}
```

排序策略：

- `cross_conflict_trace` 优先保留 `contrasts_with`、`refutes`、`qualifies` 边。
- `cross_synthesis` 优先保留多 KB 覆盖和双侧证据。
- `cross_source_verification` 优先保留最短 verified child path 与最直接 source spans。

### 10.4 verify_evidence.py 扩展

新增检查：

```text
verify_cross_path_grounding
verify_child_path_grounding
verify_cross_edge_status
verify_two_sided_evidence
verify_conflict_preserved
verify_no_payload_as_cross_evidence
```

兼容策略：

- `answer.scope != cross_kb` 时走原单库验证。
- `answer.scope == cross_kb` 时启用跨库验证。

## 11. Schema 设计

新增 schema 均遵循现有宽松风格：`additionalProperties: true`，但定义关键字段，避免过度束缚 MVP。

建议优先新增：

```text
arrange_manifest.schema.json
kb_registry.schema.json
alignment_candidate.schema.json
child_query_task.schema.json
cross_node.schema.json
cross_edge.schema.json
cross_evidence_path.schema.json
arrange_session.schema.json
cross_route_session.schema.json
```

MVP 校验可先由 `validate_arrange.py` 做程序化检查，不依赖完整 JSON Schema 校验器。

## 12. 质量门槛

`arrange_skeleton` 必须满足：

```text
required_files_exist = true
valid_child_kbs >= 2
child_query_tasks_generated > 0
cross_route_index_exists = true
verified_cross_edges 可以为 0
artifact_status = partial
```

`arrange_mvp` 必须满足：

```text
required_files_exist = true
valid_child_kbs >= 2
verified_cross_edges >= 1
verified_cross_edges_with_two_sided_evidence / verified_cross_edges >= 0.95
same_as_edges_with_strong_alignment_evidence / same_as_edges = 1.0
cross_edges_without_child_path / cross_edges <= 0.05
rejected_alignment_logged / rejected_alignment_candidates >= 0.95
book_level_only_cross_edges = 0
cross_edges_with_object_level_target / total_cross_edges >= 0.95
payload_used_as_cross_evidence = 0
```

如果 `verified_cross_edges = 0`，只能通过 `arrange_skeleton`，不能声称完成 `arrange_mvp` 或成熟跨库编排；必须报告为 `partial`，并输出 `repair_suggestions.jsonl`。

## 13. MVP 分期

### 13.1 Arrange Skeleton

目标：形成跨库编排骨架和可审计的 child query task 队列。

包括：

- KB 发现与验证。
- 候选关系生成。
- `child_query_tasks.jsonl`。
- stub 或 partial cross route index。
- `candidate_generation_report.json`。
- `arrange_validation_report.json`。

允许：

- `verified_cross_edges = 0`。
- `artifact_status = partial`。
- 大部分候选为 `needs_review`。

不允许：

- 把 skeleton 报告为完整 verified arrange。
- 用 child query task 冒充 child route session。

### 13.2 Arrange MVP

目标：在 skeleton 基础上形成少量但真实可验证的跨库关系闭环。

包括：

- KB 发现与验证。
- 候选关系生成。
- child query task 生成。
- verified / one_sided / hypothesis / rejected 状态闭环。
- cross edge 和 cross evidence path 文件。
- cross route index SQLite。
- validate_arrange 报告。
- Read Mode 可读取 cross index 并返回 cross paths。

允许：

- `cross_communities.jsonl` 为 `stubbed`。
- 部分候选为 `needs_review`。
- 部分 child query 需要人工或后续 agent 完成，但至少一部分高优先级 child queries 已执行并形成 child verified paths。

不允许：

- 用候选冒充 verified edge。
- 用 summary/payload 冒充 evidence。
- 用单侧证据写成双侧比较结论。

### 13.3 Arrange Mature

目标：支持大规模、多领域、多轮增量编排。

新增：

- 增量 add_new_kb。
- 跨库 community detection。
- 更完整的 conflict map。
- 跨图像/表格证据路径。
- 更强的 cross route scoring。
- 多 agent child query 验证。

## 14. 失败模式与防护

### 14.1 书籍节点化

防护：

- `validate_arrange.py` 检查 cross edge 的 source/target object 不应只是 book-level object。
- 质量门槛要求 `book_level_only_cross_edges = 0`。
- `cross_edge.source_object.type` 和 `target_object.type` 必须来自 concept、claim、route_node、atom、path 等。

### 14.2 标签合并

防护：

- `same_as` 必须有 child evidence paths。
- 只有 label match 时降级为 `near_equivalent`、`overlaps_with` 或 `needs_child_query`。

### 14.3 分歧被消解

防护：

- 对 `refutes`、`contrasts_with`、`qualifies` 强制记录双方立场与证据路径。
- cross answer verification 检查 conflict 是否被保留。

### 14.4 摘要替代证据

防护：

- verified cross answer 必须有 cross evidence path。
- cross evidence path 必须包含 child verified path。
- `payload_used_as_cross_evidence = 0`。

### 14.5 复制全部子图

防护：

- cross index 只保存 child references。
- `validate_arrange.py` 检查 cross artifact 中 child atom/node/edge 大规模原样复制的风险。

## 17. 验收标准

完成后应满足：

- `SKILL.md` 能明确区分 Construct / Read / Arrange 三种模式。
- Arrange Mode reference 被纳入 skill 内部，skill 可独立移植。
- 新增脚本均可通过 `python scripts/<name>.py --help` 查看参数。
- Arrange Mode 对两个以上 child KB 生成 `arrange_output/`。
- `validate_arrange.py` 能发现缺失证据、单侧证据冒充双侧证据、`same_as` 证据不足等问题。
- 原有 Construct Mode 和单库 Read Mode 不被破坏。
- cross_kb Read Mode 输出包含 cross evidence path、child paths、source spans 和 uncertainty。

## 18. 风险与取舍

### 风险 1：自动候选过多

缓解：

- MVP 中限制候选数量。
- 按 relation type、candidate type、KB pair 做分组和预算。
- 低置信候选写入 `needs_review`，不进入 verified edge。

### 风险 2：child KB Read Mode 质量不足

缓解：

- verified cross-edge 依赖 child verified path。
- child query 失败时降级，不强行生成 verified edge。
- repair suggestions 指向具体 child KB 与对象。

### 风险 3：Read Mode 复杂度上升

缓解：

- `--scope single_kb` 保持旧路径。
- `--scope cross_kb` 单独分支。
- Cross Read 的索引和 session 文件与单库 session 分离。

### 风险 4：schema 一次性设计过重

缓解：

- schema 保持关键字段 + `additionalProperties: true`。
- 质量门槛由 `validate_arrange.py` 执行，而不是过度依赖静态 schema。

## 19. 推荐最小实现边界

如果需要快速落地，第一版只做以下闭环：

1. `arrange_protocol.md`
2. `discover_kbs.py`
3. `build_alignment_candidates.py`
4. `verify_cross_edges.py`
5. `build_cross_route_index.py`
6. `validate_arrange.py`
7. `query_route.py --scope cross_kb`

其中 `query_child_kbs.py` 第一版可在 `arrange_skeleton` 中只生成 child query tasks 与 `needs_review`，不强制自动跑完所有子库查询。但 `arrange_mvp` 不能声称有 verified cross-edge，除非至少部分 child queries 已执行并产生双侧 child verified paths。这样可以先让工程骨架稳定，再逐步增强语义验证自动化。
